# 右栏顶部导出当前文章 Markdown 设计

## 背景

当前阅读页右栏顶部已有桌面工具栏，承载收藏、抓取全文、翻译、生成摘要和打开设置等动作。用户希望在同一位置新增一个导出入口，用于将当前文章以 Markdown 文件形式直接下载。

本次需求明确约束如下：

- 入口位于右栏顶部工具栏
- 导出对象为当前选中的文章
- 导出行为为直接下载 `.md` 文件
- 导出内容以文章原始正文为准，不跟随当前阅读态的翻译、双语或摘要展示
- 默认附带基础元信息：标题、发布时间、原文链接

## 目标

- 在桌面阅读页提供一个一致、低摩擦的导出入口
- 以尽量稳定的 Markdown 结构导出当前文章
- 不引入新的后端接口、全局状态或设置项

## 非目标

- 不实现批量导出
- 不新增导出格式选择
- 不导出当前翻译态、双语态或 AI 摘要态
- 不为本次需求新增服务端导出接口
- 不追求将复杂 HTML 样式无损还原为 Markdown

## 方案对比

### 方案 A：前端本地导出

在阅读页前端直接将 `article.content` 转成 Markdown，拼接元信息后生成 `Blob` 并触发浏览器下载。

优点：

- 改动范围最小，完全贴合当前“导出当前文章”的单点需求
- 不增加 API、权限边界或服务端负担
- 交互直接，点击即可完成下载

缺点：

- HTML 到 Markdown 的转换需要前端自行维护规则
- 对极复杂 HTML 结构只能做到语义优先的合理退化

### 方案 B：服务端导出接口

新增文章导出接口，由服务端读取文章并返回 Markdown，前端按钮只负责下载。

优点：

- 转换逻辑集中，后续若做批量导出可复用

缺点：

- 对当前需求属于过度设计
- 需要新增接口、测试和错误链路

### 方案 C：粗略纯文本导出

不做严格 HTML 到 Markdown 转换，只导出清洗后的文本。

优点：

- 实现最快

缺点：

- 链接、列表、图片、引用等结构损失较大
- 与“导出 Markdown”预期不匹配

## 结论

采用方案 A：在前端本地完成 Markdown 组装和文件下载。

## 架构设计

本次变更仅触及阅读页前端，保持现有数据流不变：

1. `ArticleView` 在桌面工具栏中新增导出按钮
2. 点击按钮后从当前选中的 `article` 读取标题、发布时间、原文链接和 `content`
3. 本地执行 HTML 到 Markdown 转换
4. 组装完整 Markdown 文本
5. 生成安全文件名并触发浏览器下载

为避免 `ArticleView` 继续膨胀，导出逻辑拆为两个小型辅助单元：

- Markdown 组装函数：负责将文章数据转成完整 `.md` 内容
- 下载触发函数：负责文件名清洗、`Blob` 创建、对象 URL 生命周期和点击下载

`ArticleView` 仅负责绑定按钮和调用上述能力，不承担转换细节。

## 组件与文件边界

### `src/features/articles/ArticleView.tsx`

- 在桌面工具栏区域新增导出按钮
- 仅在存在当前文章时显示该按钮
- 点击时调用导出辅助函数

### `src/features/articles/*` 新增导出辅助模块

建议新增一个与 `ArticleView` 同目录的辅助模块，承载：

- HTML 到 Markdown 转换逻辑
- Markdown 文本模板拼接
- 文件名清洗和下载触发

这样可避免把格式化与 DOM 下载细节写进组件主体，也便于单元测试直接覆盖纯函数。

## 数据来源与导出内容

导出内容固定采用“原始正文导出”策略，不跟随 UI 展示态变化：

- 标题：使用当前文章标题
- 发布时间：使用文章发布时间，输出为用户可读文本
- 原文链接：使用文章链接
- 正文：使用 `article.content`

Markdown 结构固定如下：

```md
# <标题>

发布时间：<时间>
原文链接：<URL>

<正文 Markdown>
```

当正文为空时，仍导出包含标题、发布时间和原文链接的文件，确保按钮行为稳定可预期。

## HTML 到 Markdown 转换规则

转换目标是“保留主要语义结构”，而不是复制阅读页样式。规则包括：

- 标题标签映射为 Markdown 标题
- 段落转为段落文本，并整理多余空白
- `a` 转为 Markdown 链接
- `strong`、`b` 转为粗体
- `em`、`i` 转为斜体
- `blockquote` 转为引用块
- `ul` / `ol` / `li` 转为列表
- `pre` / `code` 转为代码块或行内代码
- `img` 转为图片语法，优先保留 `alt` 和 `src`
- `hr` 转为分隔线

对于复杂嵌套或纯展示型结构：

- 忽略 `class`、`style` 等视觉属性
- 对难以表达的节点退化为合理的纯文本
- 避免直接把脏 HTML 原样写入 `.md`

仓库当前没有现成的 HTML 到 Markdown 依赖，因此实现需控制在本需求的常见语义范围内。

## 交互设计

- 按钮放在现有桌面工具栏内，与其他 `ReaderToolbarIconButton` 保持一致
- 无需弹窗或二次确认
- 点击后立即下载 `.md` 文件
- 当前无文章时不显示该按钮
- 不引入额外 loading 状态，保持交互轻量

## 文件名策略

文件名使用文章标题并追加 `.md` 后缀。

需要做安全清洗：

- 去除或替换文件系统非法字符
- 清理首尾空白
- 在标题为空时回退到稳定默认名，例如 `article.md`

## 错误处理

采用“尽量导出，失败提示”的策略：

- 标题缺失：回退默认文件名
- 正文缺失：仍导出元信息文件
- 下载链路异常：提示用户导出失败

该流程不修改文章数据，不更新 store，也不依赖后端接口，因此失败面主要集中在浏览器下载能力和本地转换逻辑。

## 测试策略

至少覆盖以下三类测试：

### 1. 按钮可见性

- 选中文章时显示导出按钮
- 无文章时不显示导出按钮

### 2. Markdown 组装

- 输出包含标题、发布时间、原文链接
- HTML 常见语义可转换为预期 Markdown
- 正文为空时仍能导出基础结构

### 3. 下载触发

- 点击导出按钮后触发文件下载
- 文件名带 `.md` 后缀
- 使用当前文章标题生成文件名

## 风险与约束

- 不同来源文章的 HTML 质量差异较大，转换结果需要接受合理退化
- 若后续用户要求“导出当前翻译态”或“批量导出”，当前方案需要重新扩展输入边界
- `ArticleView` 已较长，本次应将导出逻辑抽离，避免继续堆积行为代码

## 验收标准

- 桌面右栏顶部出现导出按钮，风格与现有工具栏一致
- 点击后可直接下载当前文章的 `.md` 文件
- 文件内容包含标题、发布时间、原文链接和由原始正文转换得到的 Markdown
- 导出不依赖后端新增接口
- 无文章时不显示导出按钮
