# 文章内图片大图预览设计

- 日期：2026-03-09
- 状态：已确认
- 需求：文章内图片可以点击查看大图

## 背景

当前阅读器正文由 [`src/features/articles/ArticleView.tsx`](../../src/features/articles/ArticleView.tsx) 直接使用 `dangerouslySetInnerHTML` 渲染文章 HTML。项目已经具备可复用的 [`Dialog`](../../src/components/ui/dialog.tsx) 组件，以及通过 [`src/server/media/rewriteHtmlImages.ts`](../../src/server/media/rewriteHtmlImages.ts) 和 [`src/app/api/media/image/route.ts`](../../src/app/api/media/image/route.ts) 维护的图片代理链路。

截至 2026-03-09，仓库中没有 `docs/summaries/` 目录可供复用经验，因此本设计以现有代码和最近的媒体修复提交为约束依据：

- `8262c8e fix(media): 修复文章图片代理与预览图加载`
- `6843f27 fix(media): 修复图片代理限流降级`

这些上下文说明新交互应尽量复用当前正文图片 URL，不应绕过既有代理和安全校验规则。

## 目标

- 允许用户点击文章正文中的图片，在站内查看单张大图。
- 同时覆盖桌面端与移动端阅读场景。
- 保持用户关闭弹层后继续在原阅读位置阅读。
- 保持实现集中在阅读器前端，不扩展后端接口和图片存储逻辑。

## 非目标

- 不做图集浏览，不支持上一张或下一张切换。
- 不提供“打开原图”“下载图片”或新的外链入口。
- 不引入独立路由或媒体详情页。
- 不改造全文抓取、HTML 清洗或图片代理的数据流程。

## 备选方案

### 方案 1：事件委托 + 站内 Dialog 单图预览

在 [`ArticleView`](../../src/features/articles/ArticleView.tsx) 的正文容器上通过事件委托识别被点击的 `img`，读取图片地址与可用文案，写入本地预览状态，再复用 [`Dialog`](../../src/components/ui/dialog.tsx) 打开大图弹层。

优点：

- 与当前 `dangerouslySetInnerHTML` 结构兼容，改动面最小。
- 不需要重写正文 HTML，也不影响沉浸式翻译拼装逻辑。
- 容易补足组件测试，适合当前“只预览单张图”的需求规模。

缺点：

- 预览态只能拿到当前 DOM 可见的信息，后续若要做图片组导航，需要再扩展状态模型。

### 方案 2：渲染前重写正文 HTML，为图片注入统一语义和包装

在渲染正文前解析 HTML，为每张图片补充 `data-*` 属性或按钮包装，再统一接管所有预览行为。

优点：

- 为后续图集导航、图片计数和更重的媒体查看器预留更系统的结构。

缺点：

- 侵入正文渲染链，容易和当前翻译渲染逻辑互相影响。
- 对本次需求明显超配，回归成本偏高。

### 方案 3：点击图片后进入独立路由或覆盖层

将图片查看器设计成更重的媒体详情流，支持路由恢复和分享。

优点：

- 扩展空间最大。

缺点：

- 对“点图看大图后关闭”这一需求明显过度设计。
- 会引入返回栈、滚动恢复和移动端手势等额外复杂度。

## 推荐方案

采用方案 1：事件委托 + 站内 `Dialog` 单图预览。

理由：

- 符合当前需求规模，最小化对文章正文渲染链的影响。
- 可以完整复用现有 `Dialog` 组件和图片代理链路。
- 风险集中在 [`ArticleView`](../../src/features/articles/ArticleView.tsx) 及其测试范围内，便于验证和回归。

## 已确认设计

### 架构与组件边界

本次改动仅发生在阅读器文章详情层，不改后端数据结构，不改文章抓取或图片代理流程。

组件边界建议如下：

- 在 [`ArticleView`](../../src/features/articles/ArticleView.tsx) 内维护轻量预览状态，至少包含是否打开、当前图片地址和可用文案。
- 新增一个轻量的图片预览展示组件，内部复用 [`Dialog`](../../src/components/ui/dialog.tsx) 负责展示大图和关闭交互。

这能把“识别用户点中了哪张图”和“如何展示大图”解耦，便于后续独立调整样式或动画。

### 数据流与交互

推荐的交互链路如下：

1. 用户点击文章正文中的图片。
2. [`ArticleView`](../../src/features/articles/ArticleView.tsx) 的正文容器通过事件委托识别命中的 `img`。
3. 从目标节点读取当前图片的展示信息，例如 `currentSrc`、`src` 和可用的 `alt`。
4. 写入本地预览状态并打开站内 `Dialog`。
5. 用户关闭弹层后回到原文章滚动位置继续阅读。

明确约束如下：

- 只响应正文区域中的 `img`，点击普通文本、按钮或普通链接不触发预览。
- 如果图片被链接包裹，点击图片本身时仍优先打开站内预览。
- 预览层只展示单张图片，不增加图组切换和附加操作区。
- 预览图片继续使用正文中已经存在的图片地址，不推导新的原图地址。

### 异常处理

异常处理以“静默降级、不影响正文阅读”为原则：

- 如果命中的图片没有有效地址，则不打开弹层。
- 如果预览图片加载失败，则保留弹层，显示简短失败提示和关闭入口。
- 不为该功能新增异步任务、后台轮询或额外网络编排。
- 即使某张图无法预览，也不能影响文章正文本身的显示与滚动。

### 可访问性

由于正文 HTML 为直出内容，原始 `img` 默认不可聚焦，因此需要做轻量增强：

- 给可预览图片补充 `tabIndex=0`。
- 为可预览图片补充 `role="button"` 与“查看大图”相关的可访问名称。
- 支持 `Enter` 和 `Space` 打开预览。
- 给图片添加明确的交互提示，例如 `cursor-zoom-in`。
- 复用 `Dialog` 现有的焦点管理与 `Esc` 关闭行为。

### 测试要求

优先通过组件测试覆盖 [`ArticleView`](../../src/features/articles/ArticleView.tsx) 的新增行为，不为本次需求引入额外的端到端复杂度。最低验证范围如下：

- 点击正文图片后可以打开预览弹层，并显示对应图片。
- 点击非图片内容不会打开弹层。
- 键盘聚焦图片后按 `Enter` 或 `Space` 可以打开预览。
- 图片被链接包裹时，点击图片仍优先打开站内预览。
- 预览图片加载失败时，会显示降级提示而不是让弹层空白失效。
- 关闭弹层后，正文仍保持可用并继续阅读。

## 实施边界

本次仅规划和实现以下交互：

- 文章正文单图大图预览
- 桌面与移动端共用同一交互模型
- 预览层失败提示与关闭行为
- 图片可聚焦和键盘触发

以下内容明确留待后续独立需求：

- 图片组导航
- 手势缩放
- 打开原图或下载
- 独立媒体路由
