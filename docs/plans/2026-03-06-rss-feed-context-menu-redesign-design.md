# RSS 源右键菜单重设计

- 日期：2026-03-06
- 状态：已确认（Approved）
- 范围：
  - `src/components/ui/context-menu.tsx`
  - `src/features/feeds/FeedList.tsx`
  - `src/features/feeds/FeedList.test.tsx`
  - `docs/plans/2026-03-06-rss-feed-context-menu-redesign-implementation-plan.md`

## 1. 背景与目标

当前 RSS 源右键菜单已经具备完整操作能力，包括：

1. `编辑`
2. `移动到分类`
3. `AI摘要配置`
4. `翻译配置`
5. `启用 / 停用`
6. `删除`

但当前实现仍然存在明显短板：

1. 菜单视觉风格接近默认 `shadcn/ui + Radix` 样式，层次、质感和品牌感不足。
2. 菜单项基本平铺，缺少对“基础编辑 / 分类归属 / 内容策略 / 风险操作”的结构化组织。
3. 子菜单、禁用态、危险操作和当前状态反馈都偏弱，操作心智不够稳定。
4. 作为高频管理入口，当前菜单没有体现信息工具型产品应有的“精致工具感”。

本次需求目标：

1. 重新设计 RSS 源右键菜单的整体视觉风格，摆脱默认组件观感。
2. 同步升级交互层，包括菜单分组、图标语义、当前分类反馈、危险操作强调和子菜单反馈。
3. 保持现有业务能力、数据流与可访问性语义不变，不引入新的数据提交链路。
4. 优先沉淀到共享 `context-menu` 组件，让后续同类菜单可以复用该设计语言。

## 2. 已确认方向与边界

### 2.1 设计方向

本次采用方向 A：**精致工具感**。

设计特征：

1. 深色半透明面板，具备克制的玻璃质感。
2. 细边框、内发光、层叠阴影和更明确的浮层层级。
3. 高级内容工具而非营销页面的审美，重在专业、耐看和高频使用体验。
4. 视觉重点落在层次、节奏、状态反馈，而不是强烈装饰。

### 2.2 已确认范围

1. 同时优化视觉层与交互层。
2. 右键菜单内部增加分区结构、图标语义和状态强化。
3. `移动到分类` 子菜单需要明确呈现“当前分类”状态。
4. `删除` 作为危险操作需要独立强调。
5. 保持 Radix 原生语义与键盘交互，不牺牲可访问性。

### 2.3 不做内容

1. 不修改业务接口与 store 数据流。
2. 不新增弹窗、搜索、输入框或创建分类能力。
3. 不重做全局 design token。
4. 不顺手重构其它菜单系统，除非它们直接复用当前共享组件并自动获益。

## 4. 方案比较与选型

### 方案 A（采纳）：升级共享 `context-menu` 视觉系统，并在 `FeedList` 重组语义结构

- 做法：把容器、条目、子菜单、分隔线等高级样式下沉到 `src/components/ui/context-menu.tsx`，同时在 `FeedList` 增加图标、分组和状态表达。
- 优点：
  - 设计语言统一，可复用，避免业务文件里堆积复杂 class。
  - 不影响现有业务逻辑，只重构表现层与信息架构。
  - 后续其它右键菜单可以直接继承收益。
- 缺点：
  - 需要谨慎设计共享组件 API，避免引入过度抽象。

### 方案 B：只在 `FeedList` 局部堆叠样式与图标

- 优点：
  - 改动表面上更集中。
- 缺点：
  - 共享组件仍然原始，复用价值低。
  - 业务层会充满菜单视觉实现细节，后续维护差。

### 方案 C：仅做视觉换肤，不调整交互层

- 优点：
  - 改动最少，风险最低。
- 缺点：
  - 无法解决当前“结构平铺、反馈弱、危险操作不突出”的核心问题。
  - 与用户明确要求的“连交互层一起调整”不符。

结论：采用方案 A。

## 5. 视觉设计

### 5.1 整体气质

右键菜单整体做成“专业内容工具的管理浮层”：

1. 采用深色、低饱和底色，避免默认浅色 popover 观感。
2. 用轻微透明度、背景模糊和内发光构建玻璃质感，但控制在克制范围内。
3. 用更深的阴影堆叠强化悬浮高度，使主菜单与子菜单形成明确层级。
4. 使用更精细的边框色和圆角比例，让菜单更接近桌面工具而非网页表单组件。

### 5.2 条目样式

1. 菜单项采用“左图标 + 主标签 + 可选尾部提示 / 状态”的结构。
2. hover 态不只改变底色，而是提供横向提亮与轻微位移感，增强可操作性。
3. 焦点态需与 hover 态一致但更稳重，保证键盘操作体验。
4. 禁用项保留足够可读性，但移除高亮反馈，体现“存在但不可操作”。

### 5.3 子菜单风格

1. 子菜单容器整体比主菜单更轻微提亮，形成上浮感。
2. `ChevronRight` 在激活时加入亮度变化或轻微位移，提示还有下一层。
3. 子菜单出现动画比默认更柔和，但应保留速度与方向感。

## 6. 信息架构与交互设计

### 6.1 菜单分区

RSS 源右键菜单调整为四个语义区：

1. 基础操作：`编辑`
2. 归类操作：`移动到分类 >`
3. 内容策略：`AI摘要配置`、`翻译配置`
4. 状态与风险：`停用 / 启用`、`删除`

分区之间通过更克制但更清晰的 separator 节奏体现，而不是纯平铺排列。

### 6.2 图标语义

建议图标如下：

1. `编辑`：`PencilLine`
2. `移动到分类`：`FolderTree`
3. `AI摘要配置`：`Sparkles`
4. `翻译配置`：`Languages`
5. `停用 / 启用`：`Power`
6. `删除`：`Trash2`

图标颜色整体保持中性，只对危险项与状态项做有限强调。

### 6.3 分类子菜单状态

1. 当前分类项保持可见但禁用。
2. 当前分类项增加更清晰状态提示，例如勾选图标或尾部 `当前` 文案。
3. `未分类` 保持同样语义。
4. 子菜单中的普通分类顺序继续遵循 `categoryMaster`。

### 6.4 危险操作与状态操作

1. `删除` 独立放在最后，并使用危险色 hover / focus 表达。
2. `停用 / 启用` 作为状态变更操作，视觉上应区别于普通配置项。
3. `删除` 不新增确认框，因为现有删除流程已经由后续对话框承接，菜单只负责入口表达。

## 7. 组件与实现策略

### 7.1 共享组件层

在 `src/components/ui/context-menu.tsx` 中：

1. 升级 `ContextMenuContent` 与 `ContextMenuSubContent` 的容器样式。
2. 升级 `ContextMenuItem` 与 `ContextMenuSubTrigger` 的交互反馈。
3. 升级 `ContextMenuSeparator` 的视觉风格，使其更像分区节奏而不是默认细线。
4. 新增轻量包装组件，避免业务层重复拼结构：
   - `ContextMenuItemIcon`
   - `ContextMenuItemLabel`
   - `ContextMenuItemHint`
   - 如有必要，增加 `destructive`、`active`、`inset` 等样式开关

这些组件应保持简单，不引入复杂抽象。

### 7.2 业务使用层

在 `src/features/feeds/FeedList.tsx` 中：

1. 使用新的菜单辅助组件组织每一项内容。
2. 为菜单项补上语义图标和必要的尾部提示。
3. 在 `移动到分类` 子菜单中标记当前分类。
4. 保持现有事件处理逻辑与 store 调用不变。

## 8. 测试策略

本次以“行为不回归 + 新状态可见”为主，不测试纯视觉 class 的每个细节。

### 8.1 需要覆盖的行为

1. 右键菜单仍可打开并进入编辑流程。
2. `移动到分类` 子菜单仍可打开并执行移动。
3. 当前分类项仍然可见，且禁用状态可识别。
4. 危险操作 `删除` 仍保留 `menuitem` 语义，避免结构调整破坏可访问性。

### 8.2 可增加的结构断言

1. 菜单项图标不要求逐一快照，但可以通过 `aria-hidden` 或测试 id 做轻量验证。
2. 如增加 `当前` 文案，可增加对应可见性断言。
3. 分区 separator 只在确有价值时断言，避免把测试绑死在纯样式实现上。

## 9. 风险与规避

1. **共享组件过度抽象风险**
   - 规避：只抽最常见的图标 / 标签 / hint 结构，不做复杂 slots 体系。
2. **视觉升级影响其他复用场景**
   - 规避：优先检查当前仓库里 `ContextMenu` 的其它调用点，如存在差异需求，用可选 class 扩展而非硬编码耦合。
3. **测试被纯样式细节绑死**
   - 规避：测试聚焦行为、可访问性角色和关键状态文案。

## 10. 验收标准

满足以下条件即可认为本次重设计完成：

1. RSS 源右键菜单视觉上明显摆脱默认组件观感，具备精致工具感。
2. 菜单信息结构更清晰，用户可快速区分基础操作、归类、策略、状态和危险操作。
3. `移动到分类` 子菜单能明确识别当前分类。
4. 现有编辑、移动分类、策略配置、启用停用和删除入口均不回归。
5. 相关单测通过，且不破坏现有菜单语义。
