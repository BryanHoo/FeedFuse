## 1. 背景与目标

当前阅读器主界面为固定三栏布局：

1. 左栏 RSS 源列表使用固定宽度 `w-60`
2. 中栏文章列表使用固定宽度 `w-[25rem]`
3. 右栏正文阅读区使用 `flex-1` 占据剩余空间

现状的问题是：

1. 左栏和中栏宽度无法根据个人阅读习惯调整
2. 宽屏设备上固定宽度利用率有限
3. 即使用户已经形成偏好，刷新或重新进入页面后也无法保留

本次需求目标：

1. 支持在桌面端拖动分割线调整左栏和中栏宽度
2. 右栏继续占据剩余宽度，不引入独立持久化宽度
3. 用户调整后的宽度需要持久化保存，并在下次打开时恢复
4. 左栏在收起后再次展开时，恢复到上次拖拽后的宽度
5. 在保持现有三栏交互和阅读性能稳定的前提下完成改动

## 2. 已确认约束

1. 仅左栏和中栏支持拖拽调宽，右栏始终吃剩余空间
2. 宽度需要持久化保存，下次打开阅读器时恢复
3. 左栏收起后再次展开，恢复上次拖拽后的宽度
4. 功能仅在桌面宽屏启用，窄屏和移动端保持现状
5. 第一版只做拖拽、边界约束、持久化和收起恢复，不扩展额外交互

## 3. 相关经验与已知约束来源

- 参考总结：[`docs/summaries/2026-03-06-middle-column-image-loading.md`](../summaries/2026-03-06-middle-column-image-loading.md)
  - 启发 1：中栏列表对无意义重渲染较敏感，拖拽过程应避免把高频宽度变化直接写入全局 store。
  - 启发 2：涉及 `ArticleList` 的交互优化应优先控制更新范围，避免为了 UI 手感牺牲中栏加载稳定性。

此外，当前代码已有一条需要保留的行为边界：

1. `sidebarCollapsed` 仍属于 `appStore` 的即时 UI 状态
2. 现有 `ReaderApp` 测试已经明确，远端持久化设置中的 `sidebarCollapsed` 不会直接回灌到 `appStore`
3. 本次新增的栏宽持久化不能顺手把 `sidebarCollapsed` 的同步语义也改掉

## 4. 方案比较与选型

### 方案 A（采纳）：保留现有 `flex` 三栏结构，手写 splitter 逻辑

- 做法：继续使用当前三栏 `flex` 布局，在左/中栏之间和中/右栏之间插入拖拽 handle，通过 `pointer` 事件调整左栏和中栏宽度，右栏继续 `flex-1`。
- 优点：
  - 改动聚焦，最大化复用现有 `ReaderLayout` 结构。
  - 不需要引入新依赖，便于控制风险与样式一致性。
  - 能直接满足“左中可调、右栏自适应”的需求。
- 缺点：
  - 需要自行实现拖拽边界、监听清理和部分可访问性细节。

### 方案 B：引入现成 split-pane 库

- 做法：把阅读器三栏改为第三方 `PanelGroup` / `Panel` / `ResizeHandle` 结构，由库处理拖拽与部分可访问性。
- 优点：
  - 基础拖拽能力和键盘交互更完整。
  - 可减少部分自研细节。
- 缺点：
  - 会引入新依赖并调整当前布局骨架。
  - 现有 `sidebarCollapsed` 语义和像素级宽度恢复逻辑需要额外适配。
  - 对当前需求范围来说偏重。

### 方案 C：改为 `CSS Grid` 三栏并自定义 track 调整

- 做法：整体切换到 `grid-template-columns`，拖拽时更新 grid track 宽度。
- 优点：
  - 三栏宽度关系表达直接。
- 缺点：
  - 需要重构现有布局样式与 `overflow` 边界。
  - 成本高于方案 A，但收益并不明显。

结论：采用方案 A，以最小布局改动实现可拖拽分栏。

## 5. 架构与状态设计

## 5.1 布局骨架

1. `ReaderLayout` 继续作为阅读器三栏装配层。
2. 左栏和中栏维持“固定像素宽度列”的角色。
3. 右栏继续使用 `flex-1` 占据剩余空间。
4. 在“左栏 / 中栏之间”与“中栏 / 右栏之间”各放一个 `ResizeHandle`。
5. 第二个 handle 只负责改变中栏宽度，右栏本身不存独立宽度。

## 5.2 状态归属

1. `sidebarCollapsed` 继续保留在 `appStore`，作为阅读器会话中的即时 UI 状态。
2. 新增持久化宽度配置，归入 `persistedSettings.general`。
3. 建议新增字段：
   - `leftPaneWidth`
   - `middlePaneWidth`
4. 默认值沿用当前体验：
   - 左栏约 `240px`
   - 中栏约 `400px`

## 5.3 更新策略

1. 拖拽过程中的临时宽度保存在 `ReaderLayout` 本地状态或 `ref` 中。
2. 拖拽结束后再把合法宽度写回 `settingsStore`。
3. 左栏收起时，视觉宽度变为 `0`，但 `leftPaneWidth` 不丢失。
4. 左栏再次展开时，直接恢复持久化中的上次宽度。

这样设计的原因：

1. 宽度属于用户偏好，适合进持久化设置。
2. 拖拽是高频交互，不适合每次位移都触发全局 store 更新。
3. 可以保持 `sidebarCollapsed` 既有语义不变，避免额外牵连 `ReaderApp` 侧的历史行为。

## 6. 交互设计与边界约束

## 6.1 拖拽行为

1. 左侧 handle 只改变左栏宽度。
2. 右侧 handle 只改变中栏宽度。
3. 使用 `pointerdown / pointermove / pointerup` 实现统一指针交互。
4. 拖拽期间给 `document.body` 施加临时 `col-resize` 光标与 `user-select: none`，结束后清理。

## 6.2 宽度边界

建议第一版使用像素级 clamp 规则：

1. 左栏最小宽度 `200px`，最大宽度 `420px`
2. 中栏最小宽度 `320px`，最大宽度 `640px`
3. 右栏保留最小安全阅读宽度 `480px`

拖拽结果需同时满足三类约束：

1. 自身栏位最小值
2. 自身栏位最大值
3. 右栏最小宽度保护

若用户继续拖拽导致右栏过窄，则按 clamp 规则停在合法边界，不继续侵占右栏。

## 6.3 桌面端开关

1. 仅在桌面断点启用拖拽交互。
2. 非桌面尺寸下不提供可交互 handle，或渲染为禁用状态。
3. 非桌面尺寸继续使用当前默认宽度与左栏收起逻辑，不新增移动端手势适配。

## 6.4 易用性与可访问性

1. handle 的视觉线可以很细，但命中区应保持 `8px~12px`，保证好拖。
2. handle 在 hover 与 dragging 时提供轻量高亮反馈。
3. 第一版不加入双击恢复默认宽度等增强交互，优先收敛范围。

## 7. 组件拆分、数据流与错误处理

## 7.1 组件拆分

1. `ReaderLayout` 负责读取 store、计算当前宽度并装配三栏。
2. 新增轻量 `ResizeHandle` 组件，负责呈现拖拽热区和基础交互入口。
3. 如实现体积需要收敛，可再抽一个小 hook，例如 `useReaderPaneWidths`，封装：
   - 拖拽起止逻辑
   - clamp 计算
   - 全局监听注册与清理

## 7.2 数据流

1. 初始化时从 `settingsStore.persistedSettings.general` 读取栏宽。
2. 若持久化值缺失或非法，则回退到默认宽度。
3. 渲染时：
   - 左栏若 `sidebarCollapsed = true`，最终渲染宽度为 `0`
   - 否则使用当前有效 `leftPaneWidth`
   - 中栏使用当前有效 `middlePaneWidth`
   - 右栏使用剩余空间
4. 拖拽过程中只更新本地临时宽度。
5. `pointerup` 后统一裁剪并写回持久化设置。

## 7.3 错误处理与清理

1. 在 `settingsSchema` 归一化阶段兜底非法宽度值。
2. 若窗口宽度不足以容纳左、中、右三栏最小值，则优先依赖桌面阈值避免进入该模式。
3. 即使在桌面模式下，也对拖拽结果做二次 clamp，始终保住右栏最小宽度。
4. 若组件在拖拽过程中卸载，必须移除全局监听和 `body` 临时样式，避免残留光标或选中文本状态。

## 8. 测试策略

## 8.1 组件测试

为 `ReaderLayout` 增加回归测试，覆盖：

1. 默认宽度渲染正确
2. 能从持久化设置读取左栏和中栏宽度
3. 左栏收起时宽度为 `0`，重新展开后恢复上次宽度
4. 拖拽左侧 handle 后写入新的左栏宽度
5. 拖拽右侧 handle 后写入新的中栏宽度
6. 超出最小/最大边界时被正确 clamp

## 8.2 设置归一化测试

补充 `settingsSchema` 或 `settingsStore` 相关测试，确保：

1. 新增栏宽字段缺失时能平滑回退默认值
2. 非法值不会污染最终渲染
3. 不会误改 `sidebarCollapsed` 现有同步边界

## 8.3 手工验收点

1. 桌面宽度下分割线可拖拽，光标与高亮反馈正确
2. 拖拽后刷新页面，左栏和中栏宽度保持
3. 左栏收起再展开，恢复上次拖拽宽度
4. 窄屏下不会出现异常拖拽体验
5. 中栏列表滚动、图片加载与右栏阅读在拖拽后无明显闪烁或卡顿

## 9. 不纳入本次范围

1. 不给右栏单独增加可持久化宽度
2. 不支持移动端手势拖拽
3. 不增加双击恢复默认宽度
4. 不做百分比宽度或多套预设宽度方案
5. 不顺带改造 `sidebarCollapsed` 的历史同步语义
