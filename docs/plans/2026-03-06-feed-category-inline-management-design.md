## 1. 背景与目标

当前分类管理已经从设置中心迁移到左栏 `FeedList`，但依然保留了单独的“管理分类”入口和分类管理弹窗。这一形态仍然把分类当成一套需要独立维护的数据，而不是围绕 RSS 源编辑自然产生的组织结构。

本次改造目标：

1. 移除左栏“管理分类”入口与独立分类管理弹窗。
2. 在添加/编辑 RSS 源时直接处理分类选择与新建。
3. 分类字段改为可输入型下拉，保存时若输入的分类不存在，则自动新建并绑定到 RSS 源。
4. 左栏分类名称支持右键菜单，直接承接重命名、上移、下移、删除。
5. 空分类不保留；当最后一个 RSS 源移走或删除时，对应分类立即自动删除。
6. 保持现有分类顺序作为 `FeedList` 与 `FeedDialog` 的共同事实来源，不退化成松散字符串模型。

## 2. 已确认约束

1. 排序通过左栏分类右键菜单中的 `上移 / 下移` 完成，不提供 `置顶 / 置底`。
2. 空分类不保留，不在左栏显示空占位分类。
3. 某分类最后一个 RSS 源移走或删除后，应立即自动删除该分类。
4. 左栏分类“编辑”通过一个小型重命名对话框完成，不做原地编辑，也不使用浏览器原生 `prompt`。
5. 在添加/编辑 RSS 源时，分类名按 `trim + lower` 归并；例如 `Tech`、` tech `、`TECH` 视为同一分类。
6. `未分类` 保持系统保底语义，不参与普通分类管理与排序。

## 4. 方案比较与选型

### 方案 A：前端串联调用现有 API

- 形态：`FeedDialog` 保存时由前端先查分类、再 `createCategory`、再 `createFeed/patchFeed`，必要时最后补一次 `deleteCategory`。
- 优点：
  - 改动面最小。
  - 现有 API 可基本不变。
- 缺点：
  - 新建分类、保存 feed、删除空分类不是原子事务。
  - 容易留下“分类创建成功但 feed 保存失败”或“feed 改了但旧空分类未删”的中间态。
  - 前端需要维护过多跨请求补偿逻辑。

### 方案 B（采纳）：服务端统一编排分类解析与清理

- 形态：保留现有分类表和排序模型，在服务端增加一层事务服务，统一处理：
  - 分类名归并
  - 新分类创建
  - feed 创建/更新
  - 旧空分类清理
  - 删除最后一个 feed 时的分类清理
- 优点：
  - 最符合“分类生命周期依赖 feed 生命周期”的业务定义。
  - 能保证数据一致性，避免中间态。
  - 前端逻辑更收敛，只关心输入与展示。
- 缺点：
  - 需要新增服务层，而不是只改 UI。

### 方案 C：彻底改成从 feeds 派生分类

- 形态：取消独立 categories 表，分类完全变成 feed 上的字符串字段。
- 优点：
  - 概念最简单。
- 缺点：
  - 会破坏现有顺序持久化、唯一约束与消费链路。
  - 变更面过大，不适合本次需求。

结论：采用方案 B。

## 5. 信息架构与交互设计

## 5.1 信息架构调整

1. `FeedList` 不再显示 `管理分类` 入口。
2. `CategoryManagerDialog` 与承载完整分类管理表格的独立路径整体移除。
3. 分类的主入口收敛为两处：
   - `AddFeedDialog` / `EditFeedDialog` 内的分类输入型下拉
   - 左栏分类名称的右键菜单

## 5.2 FeedDialog 分类字段

1. 分类字段从普通 `Select` 改为“可输入型下拉”。
2. 交互规则：
   - 展示现有分类候选，顺序继续遵循 store 中的分类顺序。
   - 允许直接输入新分类名。
   - 命中已有分类时，视为绑定已有分类。
   - 未命中时，不在输入时创建；只在点击保存后按需新建分类。
   - 选择 `未分类` 时，保存 `null` 语义，不创建普通分类。
3. `AddFeedDialog` / `EditFeedDialog` 继续只作为 `FeedDialog` 薄包装，不承载额外分类管理能力。

## 5.3 左栏分类右键菜单

1. 分类标题改为 `ContextMenu` 触发目标。
2. 菜单项固定为：
   - `编辑`
   - `上移`
   - `下移`
   - `删除`
3. `编辑` 打开一个小型重命名对话框，只修改分类名称。
4. `上移 / 下移` 复用现有 `reorderCategories` 链路，通过相邻交换实现。
5. 第一项禁用 `上移`，最后一项禁用 `下移`。
6. `未分类` 不出现普通管理菜单，也不参与移动。

## 5.4 空分类策略

1. 普通分类不允许以空占位形式存在。
2. 左栏只显示：
   - 含有 feed 的普通分类
   - 系统保底分类 `未分类`
3. 某普通分类的最后一个 feed 被移走或删除时，该分类应立即被清理并从左栏消失。
4. 新分类只有在 feed 保存成功并完成绑定后才算真正存在。

## 6. 数据流与事务边界

## 6.1 设计原则

本次需求的关键不在 UI，而在“分类生命周期被绑定到 feed 保存结果”。因此，必须将“解析分类输入、创建或复用分类、保存 feed、清理空分类”收敛到统一事务中，而不是由前端串多次请求补偿。

## 6.2 服务层职责

建议新增一层服务，例如 `src/server/services/feedCategoryLifecycleService.ts`，提供三个动作：

1. `createFeedWithCategoryResolution(...)`
2. `updateFeedWithCategoryResolution(...)`
3. `deleteFeedAndCleanupCategory(...)`

这些服务负责封装跨表规则，路由只做参数解析与错误映射。

## 6.3 添加/编辑 RSS 源

`FeedDialog` 保存时，提交分类的最终语义而不是只提交 `categoryId`：

1. 若选中已有分类，提交 `categoryId`。
2. 若输入新分类名且未命中已有分类，提交规范化后的 `categoryName`。
3. 若选择 `未分类`，提交 `null`。

服务端事务顺序：

1. 规范化分类输入，按 `trim + lower` 做归并判断。
2. 若传入 `categoryId`，直接绑定该分类。
3. 若传入 `categoryName`：
   - 先按不区分大小写查现有分类。
   - 存在则复用。
   - 不存在则创建新分类。
4. 新分类默认追加到普通分类末尾，避免破坏现有顺序语义。
5. 创建或更新 feed。
6. 若是编辑场景，还要检查旧分类是否已无剩余 feed；若挪空，则立即删除旧分类。

## 6.4 删除 RSS 源

删除 feed 时，服务层应在一个事务中完成：

1. 读取该 feed 当前所属分类。
2. 删除 feed。
3. 若原分类已无剩余 feed，则立即删除该分类。

这样可确保“删除最后一个 feed 后分类立即消失”的规则在后端一致生效。

## 6.5 显式删除分类

左栏右键菜单中的 `删除分类` 保持当前语义：

1. 删除分类本身。
2. 该分类下的 feeds 解绑到 `未分类`。

这与“空分类自动删除”不是同一规则：

- 自动删除：分类自然变空时由系统清理。
- 显式删除：用户主动移除分类定义，feeds 保留但失去普通分类绑定。

## 6.6 客户端同步策略

当前 `addFeed/updateFeed/removeFeed` 只直接更新本地 feed 集合，无法可靠同步“新建分类加入列表”或“旧空分类被删除”。因此：

1. 任一可能改变分类集合的动作成功后，应刷新一次快照或等价的分类/feeds 同步接口。
2. `snapshot.categories` 继续作为分类集合的最终事实来源。
3. 前端只做最小必要的局部更新，不手工推导分类增删，避免重复维护规则。

## 7. 错误处理与用户反馈

1. Feed 保存时：
   - 若输入分类命中已有分类并成功归并，应直接视为成功，不提示冲突。
   - 若分类解析或创建失败，则整个保存失败，不留下半成品分类。
2. 分类重命名：
   - 若名称冲突，提示“分类已存在”，对话框保持打开并保留输入值。
   - 不做隐式合并，避免用户误解现有 feeds 的归属变化。
3. 分类上移/下移：
   - 保存失败则回滚顺序并提示错误。
4. 分类删除：
   - 删除失败不提前修改本地状态。
5. 分类相关操作成功后：
   - 以快照刷新结果为准更新左栏和弹窗中的分类列表。

## 8. 测试策略

## 8.1 FeedDialog 相关测试

1. 分类字段展示现有分类候选且顺序遵循 store。
2. 输入新分类名后保存，走“新分类”提交语义。
3. 输入 ` tech ` 等大小写/空格变体时，归并到已有 `Tech`。
4. 编辑 feed 切换分类后，若旧分类被挪空，则最终不再出现在左栏。

## 8.2 FeedList 相关测试

1. 不再显示 `管理分类` 入口。
2. 分类标题支持右键菜单。
3. 右键菜单的 `编辑` 会打开小型重命名对话框。
4. `上移 / 下移` 调用分类重排并更新顺序。
5. `删除分类` 后，该分类下 feeds 回到 `未分类`。

## 8.3 服务与 API 测试

1. 新分类名可被创建并绑定到 feed。
2. 同名大小写变体归并到已有分类。
3. 编辑 feed 后旧分类变空会自动删除。
4. 删除最后一个 feed 后对应分类自动删除。
5. 任一步失败时事务回滚，不留下孤立分类或错误的分类绑定。

## 8.4 回归测试

1. `FeedList` 分类顺序仍遵循分类 position。
2. `AddFeedDialog` / `EditFeedDialog` 的分类候选顺序仍遵循分类顺序。
3. `未分类` 仍保持系统保底分组语义。

## 9. 验收标准

1. 左栏不再出现独立的 `管理分类` 入口。
2. 添加/编辑 RSS 源时，分类字段支持输入新分类并在保存时自动新建。
3. 左栏分类支持右键菜单中的编辑、上移、下移、删除。
4. 空分类不会保留；最后一个 feed 移走或删除后，对应分类立即消失。
5. 分类顺序继续影响左栏展示与 feed 弹窗中的分类候选顺序。
6. 删除分类后的 feeds 继续归入 `未分类`。
7. 相关单元测试通过，`lint` 通过。

## 10. 非目标

1. 本期不把分类改造成完全从 feeds 派生的字符串模型。
2. 本期不引入分类搜索、批量删除、分页。
3. 本期不在 `FeedDialog` 中嵌入完整分类管理面板。
4. 本期不清理 `PersistedSettings.categories` 历史字段。
