# 设计文档：全量图片强制代理服务

## 背景

- 现状：reader 会把文章正文 HTML 中的图片与中栏卡片预览图直接作为外链加载。
- 已确认的示例问题：部分源站会基于跨站 `Referer` 做防盗链或重定向，导致浏览器从 reader 页面直接请求图片时失败。
- 这类问题不只影响右栏正文，也会影响中栏 `previewImage` 的预加载与展示。

## 目标

1. 让 reader 中的全部图片统一走站内代理，不再由浏览器直接请求外站图片。
2. 保持现有正文与中栏的显示契约，不重新引入图片闪动、先出现后消失、翻译模式图片丢失等回归。
3. 提供受控、安全、可缓存的图片代理实现，而不是开放代理。

## 非目标

1. 不在本次改造中下载图片并持久化到本地磁盘或对象存储。
2. 不在数据库中永久回写代理后的图片 URL。
3. 不引入用户级开关；首版采用强制代理全部图片。
4. 不在首版引入按域名定制 `Referer`、Cookie 或复杂反爬策略。

## 相关背景与既有约束

- 相关总结：[`docs/summaries/2026-03-05-translation-preserve-html-structure.md`](../summaries/2026-03-05-translation-preserve-html-structure.md)
  - 启发：翻译模式必须保持原 HTML 结构与图片原位，不能通过重建段落列表的方式改写正文。
- 相关总结：[`docs/summaries/2026-03-06-middle-column-image-loading.md`](../summaries/2026-03-06-middle-column-image-loading.md)
  - 启发：中栏图片相关体验回归很敏感，必须保留“预加载成功后再渲染”的现有契约，避免重新引入闪动与无意义重试。

## 现有实现概览

- 入库阶段使用 `sanitizeContent` 规范化 `img[src/srcset]`，但不会代理图片 URL。
- worker 抓取 RSS 后直接把清洗后的 HTML 存为文章内容。
- `ArticleView` 直接渲染文章 HTML。
- `ArticleList` 直接使用 `previewImage` 做 `new Image()` 预加载与 `<img>` 渲染。

这意味着当前项目缺少统一的“图片交付层”。图片 URL 一旦进入前端，就会由浏览器直接访问外站。

## 方案比较

### 方案 A：读取时改写为站内代理 URL（采纳）

- 数据库继续保存原始外链。
- 在 API 输出阶段统一把正文 HTML 中的 `img[src/srcset]` 与中栏 `previewImage` 改写为站内代理 URL。
- 前端继续消费“可直接显示的图片 URL”，不参与签名与改写。

**优点**

- 不污染存量数据；未来切换签名算法、代理参数、缓存策略时无需回填数据库。
- 能统一覆盖正文图与中栏图，且最容易保持现有渲染契约。
- 代理逻辑集中在 server 侧，便于测试与安全控制。

**缺点**

- 需要新增一层图片 URL 改写逻辑，并接入文章详情与 snapshot 两个输出面。

### 方案 B：入库时永久改写图片 URL

- 在 RSS / fulltext 清洗时直接把图片改成代理 URL 并存库。

**优点**

- 读取和前端逻辑最简单。

**缺点**

- 把存储内容与交付策略耦死；后续修改签名、代理路径、域名或参数都需要回填历史文章。
- 不利于对外链内容保留原始事实来源。

### 方案 C：抓取并镜像到本地存储

- 后台下载所有图片，替换为本地静态资源地址。

**优点**

- 稳定性最好，不再依赖源站图片服务。

**缺点**

- 复杂度明显超出本次范围，需要额外处理存储、去重、清理、失败重试、容量与版权风险。

## 决策

- 采纳 **方案 A：读取时改写为站内代理 URL**。
- 全部图片强制代理，首版不保留“直连外链”的普通路径。

## 架构设计

### 分层原则

1. **存储层保留原始图片 URL**
   - 数据库中继续保存清洗后但未代理的原始 `src`、`srcset`、`previewImage`。
2. **输出层统一改写图片 URL**
   - 文章详情 API 返回正文前改写 HTML。
   - reader snapshot 返回文章列表前改写 `previewImage`。
3. **交付层统一代理远端图片**
   - 浏览器只请求站内图片代理 route。
   - 代理 route 拉取远端图片、执行安全校验、应用缓存与响应头控制。

### 模块拆分

建议新增以下模块：

1. `imageProxyUrl`
   - 输入原始外链 URL，输出带签名的站内代理 URL。
   - 作为正文图与中栏图共用的唯一 builder。

2. `rewriteHtmlImages`
   - 只改写 HTML 中 `img[src]` 与 `img[srcset]`。
   - 严格保持现有节点结构与属性顺序语义，不做正文结构重建。

3. `mediaProxyGuard`
   - 专门服务于媒体代理的 URL 安全校验。
   - 与现有 feed/fulltext 抓取 guard 分离，策略更严格。

4. `image proxy route`
   - 校验签名。
   - 校验目标 URL 是否安全。
   - 拉取远端图片并处理受控重定向。
   - 仅返回 `image/*` 响应。

## 数据流

### 正文图片

1. RSS / fulltext 抓取阶段照旧清洗并保存原始图片 URL。
2. 文章详情 API 读取文章数据。
3. server 侧对 `contentHtml` / `contentFullHtml` 执行图片 URL 改写：
   - `src` 改成站内代理 URL。
   - `srcset` 中每个候选 URL 改成站内代理 URL。
4. `ArticleView` 继续直接渲染返回的 HTML，不感知代理细节。

### 中栏图片

1. snapshot 服务返回文章列表时，统一改写 `previewImage`。
2. 如果某条数据使用从正文中提取的首图作为预览图，也必须先经过同一个 URL builder。
3. `ArticleList` 继续基于改写后的 URL 做预加载与渲染，保持现有 `ready / failed` 语义。

### 浏览器请求链路

1. 浏览器请求站内代理 URL。
2. 代理 route 校验签名与 URL 安全性。
3. 代理 route 请求远端图片资源。
4. 代理 route 返回受控图片响应与缓存头。
5. 浏览器只与本站通信，不再直连 RSS 源站图片。

## 代理接口设计

### Route 形态

- 使用单一图片代理路由，例如：`/api/media/image`
- 参数最小化：
  - `url`：原始图片 URL
  - `sig`：服务端签名

### 签名设计

- 所有代理 URL 必须带服务端签名。
- 前端不参与签名生成，只消费已签名 URL。
- 签名目标：防止外部把本站代理 route 当成公开代理滥用。
- 签名密钥由服务端环境变量提供。

## 安全边界

### URL 校验

代理 guard 必须比现有 feed 抓取 guard 更严格：

1. 只允许 `http:` / `https:`。
2. 拒绝带用户名密码的 URL。
3. 拒绝 `localhost`、loopback、private network、link-local、`.local`、`0.0.0.0` 等目标。
4. 解析域名后，要求所有 A/AAAA 记录都通过公网校验。
5. 每次重定向后的目标 URL 都必须重新通过相同校验。

### 为什么不能直接复用现有 guard

- 当前 `ssrfGuard` 为 feed/fulltext 抓取保留了 loopback 放行路径；这对于图片代理是不安全的。
- 因此必须新增独立的 `mediaProxyGuard`，避免把媒体代理做成 SSRF 通道。

### 上游请求头策略

- 不透传浏览器原始请求头。
- 不透传 cookies、authorization 等敏感头。
- 使用固定 `User-Agent`。
- 首版不开放自定义 `Referer` / Cookie / Header 能力。

### 响应校验

- 只允许返回 `image/*`。
- 非图片类型直接拒绝。
- 限制最大响应体积，防止超大资源拖垮服务。
- 限制连接、读取与总超时时间。
- 限制最大重定向次数。

## 缓存策略

### 首版原则

1. 优先支持 HTTP 缓存头透传与本地受控覆盖。
2. 成功请求允许中短期缓存。
3. 失败请求不做长缓存，避免临时失败长时间污染页面体验。

### 缓存细则

- 对成功图片返回可缓存 header。
- 如源站返回 `Cache-Control` / `ETag` / `Last-Modified`，优先在安全范围内复用。
- 对 4xx 可做极短缓存，防止页面短时间内疯狂重试。
- 对 5xx / 超时默认不做长缓存。

## 错误处理

### 正文图片

- 图片代理失败时，正文保留浏览器原生图片失败行为。
- 不插入额外错误占位或提示块，避免破坏阅读流与正文布局。

### 中栏图片

- 保持现有契约：预加载失败则记为 `failed`，不渲染图片。
- 不重新引入“先显示再消失”的状态跳变。

### 代理 route

- 签名错误、非法 URL、非图片响应返回明确的 4xx。
- 上游超时、重定向超限、上游 5xx 返回 5xx。
- 不把失败请求重定向回原始 URL。

## 对现有功能的兼容要求

### 翻译模式

- 改写只发生在 `img` 属性值层，不改变 DOM 结构。
- 必须保持图片在翻译模式下原位显示，不影响原有段落增强渲染机制。

### 中栏图片加载

- 保持现有“预加载成功后才渲染”的契约。
- 仅替换 URL 来源，不改动激活、队列、失败缓存等现有模型。

### 全文抓取与 AI 处理

- fulltext 抓取照旧保存原始 HTML。
- AI 翻译 / 摘要仍基于原始文章内容工作，不与代理 URL 耦合。

## 测试策略

### 单元测试

1. `imageProxyUrl`
   - 为同一原始 URL 生成稳定的代理 URL。
   - 篡改 `url` 后签名校验失败。

2. `rewriteHtmlImages`
   - 能改写 `img[src]`。
   - 能改写 `img[srcset]` 每个候选 URL。
   - 不改动非图片节点结构。

3. `mediaProxyGuard`
   - 拒绝 private / loopback / `.local` / 凭据 URL。
   - 接受合法公网图片 URL。

### Route 测试

1. 合法签名 + 合法图片响应 → `200`。
2. 上游返回非图片类型 → 拒绝。
3. 上游 302 到公网图片 → 允许。
4. 上游重定向到内网地址 → 拒绝。
5. 超时 / 重定向过多 → 失败。

### UI 回归测试

1. 文章详情 API 返回的 HTML 中，图片被改写为代理 URL。
2. `ArticleView` 渲染代理后的图片时，翻译模式仍保持图片原位。
3. `ArticleList` 预加载和渲染均使用代理 URL。
4. `ArticleList` 失败缓存与不闪动契约保持不变。

## 验收标准

1. reader 中的正文图与中栏图全部走站内代理。
2. 已知防盗链案例能稳定显示图片。
3. 不破坏翻译模式图片原位与中栏图片加载体验。
4. 代理 route 不是开放代理，且对内网/loopback 目标有明确防护。
5. 相关单元测试、route 测试与 UI 回归测试可覆盖上述关键行为。

## 实施建议

1. 先实现 URL builder、签名与 guard。
2. 再实现 proxy route 与 route 级测试。
3. 然后接入文章详情与 snapshot 输出改写。
4. 最后补 UI 回归测试，确保正文图与中栏图统一通过代理路径工作。
